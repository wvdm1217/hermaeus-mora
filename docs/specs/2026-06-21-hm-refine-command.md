# hm refine <entry_id> Command

## 1. Overview
The `hm refine <entry_id>` command introduces agentic processing to the Hermaeus Mora system. It takes a raw human-authored Markdown entry, passes it to a Large Language Model (LLM), and automatically generates a summary and extracts relevant tags. Furthermore, it extracts structured knowledge graph components (Entities, Relations, and Observations) and populates an agent memory system utilizing Markdown files stored in specific directories (`entities/`, `relations/`, `observations/`).

## 2. Motivation & Goals
- **Why are we doing this?** Raw notes often lack structured metadata and explicit connections. By automating the extraction of summaries, tags, and knowledge graph components, we build a powerful, interlinked context for the agent without requiring manual curation from the user.
- **Goals:**
  - Automatically update the frontmatter of the specified human note with a generated `summary` and `tags`.
  - Extract knowledge graph components based on the MCP standard into a Markdown-based memory system.
  - Store these components in dedicated agent memory directories (`data/agent_memory/entities/`, `data/agent_memory/relations/`, `data/agent_memory/observations/`).
  - Set the `author` metadata of all generated agent memory files to `hermaeus_mora`.
  - Link extracted memories to their source note via a `source_entry_id` foreign key where applicable (not all agent memories need to be linked to an entry, but those generated by `hm refine` will be).
  - Integrate these agent memory files into the hybrid search system, splitting the results to distinctly display human notes versus agent notes without arbitrary weighting.
- **Non-Goals:**
  - Modifying the core body content of the user's original Markdown note.
  - Implementing a separate graph database (all graph data remains as plain text Markdown).

## 3. Architecture & Design
How does this feature fit into the existing system?
- **Components involved:**
  - **CLI (`cli.py`):** A new `refine` command that accepts an `<entry_id>`.
  - **LLM Integration:** A provider-agnostic module that interfaces with an LLM. It will use Gemini by default and also support Ollama. It will heavily utilize structured output schemas (e.g., Pydantic) for reliable extraction.
  - **Storage (`storage.py`):** Functions to update the existing entry's frontmatter and to create/update agent memory files in the `data/agent_memory/` directory. Handles deletion of existing `source_entry_id` linked files on re-run.
  - **Search (`search.py`):** Enhancements to the hybrid search (DuckDB/Vector) to index these new agent memory directories and split the search results into human-authored and agent-authored lists.
- **Data flow:**
  1. The user executes `hm refine <entry_id>`.
  2. The system reads the target Markdown note.
  3. The LLM processes the content and returns structured JSON containing a summary, tags, entities, relations, and observations.
  4. The system updates the source note's YAML frontmatter.
  5. Before creating new files, the system finds and deletes any existing agent memories in `data/agent_memory/` that have a `source_entry_id` matching the target `<entry_id>`.
  6. The system generates new Markdown files in `data/agent_memory/entities/`, `data/agent_memory/relations/`, and `data/agent_memory/observations/` with `author: hermaeus_mora` and `source_entry_id: <entry_id>`.
  7. The search index ingests the updated source note and the new agent memory files.
- **Dependencies:**
  - An LLM provider interface.
  - Structured output parsing capabilities (e.g., Pydantic).

## 4. Implementation Details
- **Data structures & models:**
  - **Agent Markdown Schema (Example):**
    - `data/agent_memory/entities/<slugified_entity_name>.md`
      ```yaml
      ---
      id: <uuid>
      type: entity
      entity_type: person
      author: hermaeus_mora
      source_entry_id: <uuid_of_source_entry>
      created_at: 2026-06-21T12:00:00Z
      ---
      # <Entity Name>
      ```
    - `data/agent_memory/observations/<uuid>.md`
      ```yaml
      ---
      id: <uuid>
      type: observation
      entity: <slugified_entity_name>
      author: hermaeus_mora
      source_entry_id: <uuid_of_source_entry>
      created_at: 2026-06-21T12:00:00Z
      ---
      <Observation Content>
      ```
    - `data/agent_memory/relations/<uuid>.md`
      ```yaml
      ---
      id: <uuid>
      type: relation
      from: <slugified_entity_name>
      to: <slugified_entity_name>
      relation_type: works_at
      author: hermaeus_mora
      source_entry_id: <uuid_of_source_entry>
      created_at: 2026-06-21T12:00:00Z
      ---
      ```
- **Core Logic / Algorithms:**
  - **Slugification:** Entity names must be slugified to ensure valid and consistent filenames.
  - **Re-run / Overwrite Logic:** When running `hm refine <entry_id>`, the system will first query for and delete any existing agent memories in `data/agent_memory/` that carry the metadata `source_entry_id: <entry_id>`. This acts as a clean slate for the entry being refined.
  - **Duplicate Handling:** Duplicates across different source entries are not explicitly handled during extraction. Future agent capabilities will interact with and manually refine/merge these memories.
  - **Frontmatter Updating:** A safe YAML parsing and dumping mechanism must be used to preserve existing frontmatter when appending the summary and tags.
- **API Contracts:**
  - The LLM prompt must strictly define the expected schema for Entities (name, entityType), Relations (from, to, relationType), and Observations (entityName, contents).

## 5. Testing & Edge Cases
- **Unit Tests:**
  - Validating the parsing of structured LLM outputs.
  - Ensuring entity name slugification handles special characters correctly.
  - Testing the correct generation and formatting of agent memory Markdown files, including the `source_entry_id`.
  - Verifying that running `hm refine` successfully deletes older memories tied to the same `source_entry_id`.
  - Verifying the search logic correctly splits results between human notes and agent notes.
- **Integration Tests:**
  - End-to-end execution of the `hm refine` command with a mocked LLM response, verifying both the source note update and the generation of memory files.
- **Edge Cases & Failure Modes:**
  - The LLM fails to return valid JSON or hallucinates required schema fields.
  - The target entry does not exist.
  - Concurrent refinement operations causing file lock issues.
  - Extracted entities contain extremely long names that violate filesystem constraints.

## 6. Observability & Reliability
- **Logs:**
  - Log the initiation of the refine process.
  - Log LLM token usage and extraction success/failure rates.
  - Log any errors encountered during file generation or frontmatter updating.
- **Metrics / Dashboards:** N/A for local CLI.
- **Alerts:** Provide clear, actionable console errors to the user if the refinement process fails.

## 7. Rollout & Rollback
- **Feature Flags:** None required.
- **Deployment Steps:** Delivered as part of the 0.4.0 CLI update.
- **Rollback Plan:** Since files are plain Markdown, users can rollback by manually deleting the generated files in the agent memory directories and reverting their source note using version control.
